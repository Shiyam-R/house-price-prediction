# House Price Prediction API

[![CI Pipeline](https://github.com/Shiyam-R/house-price-prediction/actions/workflows/ci.yml/badge.svg)](https://github.com/Shiyam-R/house-price-prediction/actions/workflows/ci.yml)

A production-style, end-to-end machine learning system that predicts house sale prices from the Ames Housing dataset — served via a FastAPI REST API, containerized with Docker, and continuously validated through a GitHub Actions CI/CD pipeline.

This project isn't just a trained model in a notebook. It's the full path from raw data to a running, testable, deployable service:

**Data → Feature Engineering → Model Training → API → Docker → CI/CD**

## Model Performance

| Metric | Value |
|---|---|
| Model | XGBoost (hyperparameter-tuned) |
| R² Score | 0.9005 |
| RMSE | $28,050 |

## Live API Capabilities

- **`POST /predict`** — accepts raw house features, returns a predicted price with a confidence range
- **`GET /health`** — service health check, used by Docker's `HEALTHCHECK` and the CI pipeline
- **`GET /docs`** — interactive Swagger UI (auto-generated by FastAPI) for exploring and testing every endpoint
- Full input validation via Pydantic — invalid categorical values or out-of-range numbers are rejected with a clear `422` error before any model computation runs

## Model Versioning

Trained model artifacts are versioned by folder, not just tracked as loose files. Each version under `artifacts/` (e.g. `v1.0.0/`) is self-contained: the trained artifacts themselves, a machine-readable `metadata.json` (version, metrics, framework versions), and a human-readable `MODEL_CARD.md` documenting intended use, features, and limitations.

The API selects which version to serve via the `MODEL_VERSION` environment variable (defaults to `v1.0.0`):
```bash
MODEL_VERSION=v1.0.0 uvicorn app.main:app --host 0.0.0.0 --port 8000
```
`/health` and `/version` read metrics directly from the loaded version's `metadata.json` — they always reflect exactly what's actually running, not a value someone has to remember to update in source code separately.

See [`artifacts/v1.0.0/MODEL_CARD.md`](artifacts/v1.0.0/MODEL_CARD.md) for full model documentation, including intended use and known limitations.

## Running Locally (without Docker)

```bash
# 1. Create and activate a virtual environment
python -m venv venv
venv\Scripts\activate        # Windows
source venv/bin/activate     # macOS/Linux

# 2. Install dependencies
pip install -r requirements.txt

# 3. Run the API
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

Visit `http://localhost:8000/docs` for the interactive API documentation.

## Running with Docker

```bash
# Build the image
docker build -t house-price-api .

# Run the container
docker run -p 8000:8000 house-price-api
```

Visit `http://localhost:8000/health` to confirm the service is running.

## CI/CD Pipeline

Every push and pull request automatically triggers a GitHub Actions pipeline that:

1. Checks out the repository
2. Sets up Python 3.13
3. Installs dependencies
4. Verifies dependency consistency (`pip check`)
5. Validates Python syntax across the codebase
6. Runs the unit test suite
7. Builds the Docker image
8. Runs the built image as a real container and polls `/health` to confirm the service actually starts and serves traffic

See `.github/workflows/ci.yml` for the full pipeline definition, and the **Actions** tab of this repository for live run history.

## Key Engineering Decisions

- **Preprocessing is fully replicated inside the API**, not just baked into the `.pkl` files — the model file alone doesn't capture how raw input needs to be transformed.
- **Multi-stage Docker build** keeps the final image free of build tools and pip caches — only the runtime virtual environment and application code ship.
- **Non-root container user** — the app never runs as root inside the container.
- **CI verifies the container actually runs**, not just that it builds — a build succeeding says nothing about whether the app starts correctly.
